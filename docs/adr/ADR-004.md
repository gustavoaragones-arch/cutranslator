# ADR-004: Two-Product Architecture — Main Tool and Offal Product

**Status: Accepted**
**Date:** 2026-06-08

## Context

The main Cutranslator dataset is keyed on muscle cuts × country pairs. Its value comes from
comparison density — what does country A call the cut that country B has a name for. The query
that users come to answer is: "I have this cut name from one country; what does another country
call it, and where do I find it at the butcher?"

Offal cuts structurally do not fit this model. Country-by-country retail translation is the
wrong query for offal. The interesting unit is cultural-prep tradition:

- Argentine achuras (parrilla sequence, parrillero craft, regional variation)
- Mexican lengua (taqueria tradition, braising practice, regional preparations)
- Nalli Nihari (Mughal court origin, overnight technique, Delhi–Karachi cross-border continuity)
- Rocky Mountain Oysters (ranching tradition, festival culture, prairie variant naming)

These are not bilateral name-translation questions. They are culinary-cultural reference
questions about how a given offal is prepared, by whom, and in what context.

Forcing offal into the main tool produced a cascade of symptoms:

- `coverage: "sparse"` flags — a bespoke field encoding "this cut doesn't fit the model"
- Section suppression logic — hiding "Countries that use it" and "Names by country" for offal
  cuts because the data asserted false coverage
- Credibility leaks — English fallback strings rendering as if they were researched translations
- Architectural friction across multiple batches (ADR-002, multiple Batch 25–28 supplements)
- Diagram handling divergence — offal overlays lived in a different SVG directory and required
  special routing logic in the diagram component

## Decision

Two products, shared canonical layer, separate everything else.

| Dimension | Main tool | Offal product |
|---|---|---|
| Route | `/cuts/[canonical]` | `/offal/cuts/[id]`, `/offal/traditions/[id]` |
| Dataset | Muscle cuts × country pairs | Offal cuts × cultural traditions |
| Primary value | Cross-country name translation | Culinary/cultural reference |
| Data model | `regionalNames`, `regionalCuts` | `OffalTradition` (lib/offalData.ts) |
| Template | Country-pair comparison tool | Cut-first + tradition-first hybrid browse |

**Shared layer:** `CanonicalId` union in `lib/types.ts`. Anatomical truth does not fork. All
10 offal canonicals remain in the union and are reachable at `/cuts/[canonical]` in the main
tool with a cross-link to their dedicated `/offal/cuts/[id]` page.

**Separate:** routes, page templates, data files, Tier 3 nodes, diagram handling, tradition data.

## Consequences

- Sparse-coverage logic retired from main tool (ADR-002 superseded by this decision)
- `CoverageDensity` type and all `coverage: "sparse"` fields removed from the codebase
- 10 offal canonicals remain in `CanonicalId` union, reachable in main tool via cross-link
- The safety-net heuristic (filtering fallback English names from `namesByCountry`) is
  retained — it is useful for all canonicals, not just offal
- ADR-005 governs forward-reference discipline for offal canonical expansion (testicles,
  sweetbreads, brain, lungs) — these are deliberate batch work within the offal product,
  not main-tool scope
- Offal tradition data lives in `lib/offalData.ts`; the `OffalTradition` type is the expansion
  point for PR 2 and beyond

## Forward-reference discipline (ADR-005 preview)

When an offal tradition references a cut that does not yet have a canonical (e.g. testicles in
Rocky Mountain Oysters, sweetbreads/mollejas in Argentine Achuras), the tradition ships with
`cuts: []` or with only the subset of cuts that have valid `CanonicalId` entries. The
`cuts[]` field is updated in the same PR that promotes the new canonical. This pattern prevents
forward references that break `tsc` and keeps the data self-consistent at every commit.
